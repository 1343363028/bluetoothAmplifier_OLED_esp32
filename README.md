# ESP32 可编程蓝牙音箱 —— 项目总结

## 一、项目背景与目标

### 1.1 项目来源

本项目源自《电气工程实践基础——Arduino 综合实践》课程的课堂任务。课程要求基于 ESP32 开发板，综合运用蓝牙通信、红外遥控、LED 灯效等技术，完成一个"可编程蓝牙音箱"作品。

### 1.2 项目目标

构建一个完整的 ESP32 蓝牙音箱系统，实现以下核心能力：

- **蓝牙音频接收**：ESP32 作为 A2DP Sink 接收手机蓝牙音频，经 MAX98357A 功放驱动扬声器播放
- **红外遥控交互**：通过红外遥控器实现播放/暂停/停止/音量/上下曲控制
- **OLED 信息显示**：实时展示播放状态、歌手、歌词、音量等信息，支持中文显示与超宽文本滚动
- **LED 灯效反馈**：不同播放状态下呈现不同灯效（彩虹/呼吸/常亮），提供直观的视觉反馈

## 二、核心功能模块

### 2.1 模块总览

| 模块 | 核心器件 | 通信方式 | 功能 |
|------|---------|---------|------|
| 蓝牙音频 | MAX98357A + ESP32 | I2S + A2DP | 接收手机蓝牙音频并输出至扬声器 |
| 红外遥控 | VS1838B | NEC 红外协议 | 遥控器控制播放/暂停/停止/音量/上下曲 |
| OLED 显示 | SSD1306 128×64 | I2C | 中文显示状态、歌手、歌词、音量 |
| LED 灯效 | WS2812B ×3 | 单总线 | 状态视觉反馈 |

### 2.2 硬件接线表

| 元件 | 引脚映射 | 电源 |
|------|---------|------|
| **MAX98357A** | LRC→GP26, BCLK→GP25, DIN→GP27 | VCC→5V, GND→GND |
| **红外接收管 1838** | OUT→GP15 | VCC→5V, GND→GND（10kΩ上拉） |
| **WS2812B** | DI→GP2 | VCC→5V, GND→GND |
| **SSD1306 OLED** | SCK→GP22, SDA→GP21 | VDD→5V, GND→GND |
| **扬声器** | 接 MAX98357A 输出端 | 4Ω/3W |

**状态机设计：**

```
state=0 (待机) ──红外播放──→ state=1 (播放)
state=1 (播放) ──红外暂停──→ state=2 (暂停)
state=1 (播放) ──红外停止──→ state=3 (停止)
state=2 (暂停) ──红外播放──→ state=1 (播放)
state=2 (暂停) ──红外停止──→ state=3 (停止)
state=3 (停止) ──红外播放──→ state=1 (播放)
```

## 三、技术架构与选型

### 3.1 硬件选型

| 选型决策 | 方案 | 选择理由 |
|---------|------|---------|
| 主控 | **ESP32 (MakePico D0WDQ6)** | 内置 Wi-Fi + 蓝牙，双核 240MHz，A2DP 硬件加速 |
| 音频功放 | **MAX98357A** | I2S 数字输入，无需 DAC，内置 Class D 放大，接线简单 |
| 红外接收 | **VS1838B** | 38kHz 载波，NEC 协议兼容，成本低廉 |
| 显示屏 | **SSD1306 OLED 128×64** | I2C 接口省引脚，U8g2 库中文支持好，对比度高 |
| LED | **WS2812B ×3** | 单线控制，RGB 全彩，独立寻址 |

### 3.2 软件依赖

| 库 | 版本/来源 | 用途 |
|----|----------|------|
| `DFRobot_MAX98357A` | DFRobot 官方 | MAX98357A 驱动 + A2DP 蓝牙接收 + AVRC CT 控制 |
| `IRremote` | by Armin Joachimmeyer | 红外 NEC 协议解码 |
| `U8g2` | by oliver (olikraus) | SSD1306 驱动 + 中文字体渲染 |
| `ESP32_WS2812_Lib` | — | WS2812B RGB 灯带控制 |

### 3.3 关键配置

- **Arduino IDE 分区方案**：必须选择 **"Huge APP (3MB No OTA/1MB SPIFFS)"**
  - 原因：默认分区仅 1.3MB 程序空间，U8g2 全量中文字体（`u8g2_font_wqy12_t_gb2312`，~500KB）+ 蓝牙协议栈编译后约 1.4MB，超出限制
  - Huge APP 方案提供 3MB 程序空间，编译后占用约 47%

- **开发板选择**：ESP32 Dev Module（对应 ESP32 Arduino Core 3.1.1，基于 ESP-IDF v5.3）

## 四、项目进化历程

### V1.0 —— 文档代码提取（初始版本）

**起点**：从教学文档的 7 张代码截图（image31~image37）中 OCR 提取完整代码。

**功能**：蓝牙 A2DP + 红外遥控 + WS2812B 灯效

**存在的问题**：
- `PinDefinitionsAndMore.h` 缺失 → 编译报错
- `TYPE_GRGB` 不存在 → 应为 `TYPE_GRB`
- `esp_avrc_tg_api.h` 在 ESP-IDF 5.3 中不存在 → TG API 已合并到 `esp_avrc_api.h`
- 灯效状态只响应红外遥控，手机端播放/暂停时灯不变
- AVRC 回调参数字段名与 ESP-IDF 5.3 不匹配（`cmd`→`key_code`，`state`→`key_state`）

### V1.1 —— 编译问题修复

**修复清单**：

| 问题 | 根因 | 解决方案 |
|------|------|---------|
| `PinDefinitionsAndMore.h: No such file or directory` | IRremote 库示例的辅助文件，非标准头文件 | 创建该文件，定义 `IR_RECEIVE_PIN=15`，加入 `#define DISABLE_IR_SEND` |
| `TYPE_GRGB was not declared` | 文档截图笔误，WS2812B 数据顺序为 GRB | 改为 `TYPE_GRB` |
| `esp_avrc_tg_api.h: No such file or directory` | ESP-IDF 5.3 将 TG API 合并到 `esp_avrc_api.h` | 删除该 include，统一用 `esp_avrc_api.h` |
| AVRC 回调参数名不匹配 | ESP-IDF 5.3 重命名了字段 | `param->psth_cmd.cmd` → `key_code`，`psth_cmd.state` → `key_state` |
| IR_SEND_PIN pragma 提示 | 项目只接收不发送红外 | `PinDefinitionsAndMore.h` 中加 `#define DISABLE_IR_SEND` |

### V2.0 —— 精简重构 + OLED 集成

**重大调整**：移除灯效代码，聚焦蓝牙+红外核心，新增 OLED 显示。

**变更原因**：多模块同时调试困难，先确保核心功能可用再加功能。

**OLED 初版方案**：
- 使用 `Adafruit_GFX` + `Adafruit_SSD1306`
- 仅显示 ASCII 文本（蓝牙状态、歌名、歌手、音量条）
- **问题**：不支持中文显示

### V2.1 —— 中文显示方案切换

**从 Adafruit 切换到 U8g2**：
- Adafruit_SSD1306 只支持 ASCII，无中文字体
- 换用 U8g2 库 + `u8g2_font_wqy12_t_gb2312` 全量中文字体

**新问题**：编译后 1.4MB > 默认分区 1.3MB → **Sketch too big**

**解决**：修改 Arduino IDE 分区方案为 "Huge APP (3MB No OTA/1MB SPIFFS)"

### V2.2 —— 中文乱码问题

**现象**：OLED 上所有中文显示为乱码

**排查过程**：
1. 首先尝试用 Unicode 转义（`\u84DD\u7259`）→ 仍乱码
2. 确认文件编码为 UTF-8 → 不是编码问题
3. **根因**：使用了 `u8g2.print()` 而非 `u8g2.drawUTF8()`
   - `print()` 是 Arduino 通用接口，把 UTF-8 多字节中文拆成单字节逐个送显 → 乱码
   - `drawUTF8()` 是 U8g2 专门为 Unicode 设计的函数，正确识别 UTF-8 多字节序列

**解决方案**：全部 `print()` 调用改为 `drawUTF8()`，中文显示正常

### V2.3 —— OLED 布局迭代

**布局演进**：

| 版本 | 第1行 | 第2行 | 第3行 | 第4行 |
|------|-------|-------|-------|-------|
| v2.3a | 状态 | 歌名-歌手 | 专辑(滚动) | — |
| v2.3b | 状态 | 歌手 | 歌词(两行拆分,滚动) | — |
| v2.3c | 状态 | 歌手 | 歌词(滚动) | 音量条 |
| **v2.3d（当前）** | **状态** | **歌手** | **歌词(滚动)** | **音量条** |

**关键发现**：用户使用的音乐 App（QQ音乐）通过 Title 字段实时推送歌词——歌名和歌词行交替出现在 Title 中。因此 Title 字段既是歌名义是歌词，无需单独获取歌词接口。

**未连接状态优化**：断开蓝牙后不再显示四行小字，改为 16px 大字号居中显示"未连接"。

**音量条重叠修复**：原方案 `barX=14` 写死，但"音量"两字在 12px 字体下实际占 24px，文字与进度条重叠。改用 `getUTF8Width()` 动态测量文字宽度，进度条紧跟其后。

### V3.0 —— LED 灯效回归

在 V2.3 的基础上重新加入 WS2812B 灯效模块，形成完整四模块系统。

灯效状态映射：
- `state=0` 待机 → 蓝色常亮
- `state=1` 播放 → 彩虹流动
- `state=2` 暂停 → 蓝色呼吸
- `state=3` 停止 → 绿色常亮

### V3.1 —— AVRC Target 绝对音量同步（尝试与回退）

**问题**：手机端调节音量时，ESP32 本地 `volume` 变量不变，OLED 音量条不更新。

**根因分析**：DFRobot 库只初始化了 AVRC CT（控制器角色），未初始化 AVRC TG（目标端角色）。手机发来的 SetAbsoluteVolume 命令无法到达 ESP32。

**尝试方案**：
- 手动调用 `esp_avrc_tg_init()` + 注册回调
- 注册 `ESP_AVRC_RN_VOLUME_CHANGE` 通知能力
- 注册 passthrough 命令过滤
- 回调中处理 `ESP_AVRC_TG_SET_ABSOLUTE_VOLUME_CMD_EVT`
- 红外调音量时通过 `esp_avrc_tg_send_rn_rsp()` 反向通知手机

**结果**：编译通过但功能不工作。原因推测为 DFRobot 库内部已完整管理蓝牙协议栈生命周期，外部手动初始化 AVRC TG 与库内部状态冲突。

**决策**：回退 AVRC Target 相关代码，音量仅由红外遥控控制，保持系统稳定性。

## 五、主要挑战与解决方案

### 5.1 文档代码为图片格式

**挑战**：教学文档中的完整项目代码以 7 张截图嵌入，无法直接复制文本。

**方案**：从 docx 中提取图片文件，逐张阅读识别代码内容，手动整理修复。

**收获**：OCR/图片识别过程中需注意代码缩进、分号、大小写等细节，整理后需逐一编译验证。

### 5.2 ESP-IDF 版本兼容性

**挑战**：文档代码基于旧版 ESP-IDF 编写，用户环境为 ESP32 Arduino Core 3.1.1（ESP-IDF v5.3），API 发生了多项变更。

**受影响项**：
| 旧 API | 新 API (ESP-IDF 5.3) |
|--------|---------------------|
| `esp_avrc_tg_api.h` | 合并到 `esp_avrc_api.h` |
| `ESP_AVRC_PSTH_FILTER_SUPPORT_CMD` | `ESP_AVRC_PSTH_FILTER_SUPPORTED_CMD` |
| `param->psth_cmd.cmd` | `param->psth_cmd.key_code` |
| `param->psth_cmd.state` | `param->psth_cmd.key_state` |
| C 复合字面量取地址 `&(type){...}` | C++ 不允许，需声明命名变量 |

**方案**：逐一查阅 ESP-IDF 5.3 头文件确认正确 API，修正所有不兼容项。

### 5.3 Flash 空间不足

**挑战**：默认分区 1.3MB 程序空间，编译后 1.4MB 超出限制。

**方案**：修改分区方案为 "Huge APP (3MB No OTA/1MB SPIFFS)"。代价是放弃了 OTA 无线升级能力，但课堂项目无需此功能。

### 5.4 OLED 中文显示

**挑战**：Adafruit_SSD1306 不支持中文；U8g2 中文乱码；全量中文字体占 Flash 大。

**解决方案链**：
1. Adafruit_SSD1306 → 换 U8g2（内置中文字体）
2. `print()` → `drawUTF8()`（正确处理 UTF-8 多字节）
3. 文件编码确保 UTF-8（避免 GBK 与 UTF-8 不匹配）
4. 分区方案改 Huge APP（容纳 ~500KB 中文字体）

### 5.5 蓝牙协议栈与 AVRC TG 冲突

**挑战**：DFRobot 库封装了完整的蓝牙 A2DP + AVRC CT 初始化流程，外部手动初始化 AVRC TG 导致功能异常。

**方案**：放弃 AVRC TG 手动初始化，音量仅由红外遥控本地控制。如需手机端音量同步，未来需深入修改 DFRobot 库源码或替换为更底层的蓝牙 API。

## 七、未来规划与优化方向

### 7.1 短期优化

| 方向 | 说明 | 难度 |
|------|------|------|
| **手机端音量同步** | 深入 DFRobot 库源码，在其蓝牙初始化流程中集成 AVRC TG，或替换为原生 ESP-IDF A2DP Sink API | 高 |
| **手机端播放状态同步** | 同上，需 AVRC TG passthrough 回调正确工作 | 高 |
| **红外遥控码自适应** | 启动时进入学习模式，自动采集遥控器按键编码，存入 NVS | 中 |
| **OLED 歌词逐行显示** | 利用 Title 字段实时推送特性，区分歌名（首次 Title）和歌词（后续 Title），分别显示 | 低 |

### 7.2 中期扩展

| 方向 | 说明 |
|------|------|
| **按键控制** | 增加物理按键作为红外遥控的补充，避免遥控器丢失后无法操作 |
| **Wi-Fi 音乐流** | 利用 ESP32 的 Wi-Fi 能力，从网络电台/本地服务器播放音乐 |
| **多音效模式** | 利用 MAX98357A 的 DSP 通道实现不同的音效预设 |
| **低功耗优化** | Light Sleep 模式，红外唤醒，延长电池续航 |

### 7.3 架构层面

| 方向 | 说明 |
|------|------|
| **FreeRTOS 多任务** | 将蓝牙、红外、OLED、LED 分配到不同任务，避免 loop() 中的 delay() 阻塞 |
| **事件驱动架构** | 用 FreeRTOS 队列/事件组替代全局变量，提升状态管理可靠性 |
| **DFRobot 库重构** | Fork 并修改库源码，在 A2DP 初始化回调中集成 AVRC TG 注册，彻底解决音量同步问题 |
| **配置文件化** | 将红外编码、引脚映射、蓝牙名称等提取为 config.h，便于不同硬件适配 |
